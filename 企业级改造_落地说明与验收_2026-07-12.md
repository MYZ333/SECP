## 一、上线前必做（顺序不能错）

1. **备份数据库**（P4 会改列并加密回填，务必先备份）：
   ```bash
   mysqldump -u root -p --default-character-set=utf8mb4 hda_db > hda_db_backup_20260712.sql
   ```
2. **执行列扩容脚本**（敏感字段密文更长，改为 TEXT）：
   ```bash
   mysql -u root -p --default-character-set=utf8mb4 < backend/src/main/resources/data_security_upgrade.sql
   ```
3. **后端装依赖**：`pom.xml` 新增了 `redisson`、`spring-boot-starter-cache`，首次启动会自动拉取。
4. **前端装依赖**（Web 端新增 `jsencrypt`）：
   ```bash
   cd web && npm install
   ```
5. **首次启动做加密迁移**：把 `application.yml` 的 `hda.crypto.migrate` 设为 `true`（或设环境变量 `HDA_CRYPTO_MIGRATE=true`）启动一次，日志出现“加密迁移完成”后**改回 false** 再正常运行。迁移是幂等的，重复跑不会二次加密。
6. **生产环境务必配置固定密钥**（否则多节点/重启会出问题）：
   - `JWT_SECRET`（≥32 字节）
   - `HDA_AES_KEY`（敏感字段 AES 密钥）
   - `RSA_PUBLIC_KEY` / `RSA_PRIVATE_KEY`（密码传输 RSA 密钥对，多节点必须固定；单机可留空自动生成）

> Redis 是所有能力（限流/登录态/黑名单/锁/幂等/缓存）的基础，启动前确保 Redis 在跑。

## 二、各模块改了什么（对应 10 项决策）

### 登录安全
- **双令牌**：登录返回 `token`(access, 30 分钟) + `refreshToken`(7 天)。access 过期时前端用 refresh 静默换新，用户无感。`/api/auth/refresh`。
- **登出黑名单**：`/api/auth/logout` 把当前 token 的 `jti` 拉入 Redis 黑名单（TTL=剩余有效期），旧 token 立即失效。
- **单点登录（挤下线）**：沿用并强化原有 Redis 登录态；被挤下线返回专门错误码 `1101`，前端弹“账号已在其他设备登录”。
- **密码 RSA 传输加密**：`/api/auth/public-key` 下发公钥，前端 `jsencrypt` 加密密码再提交，后端私钥解密。**向后兼容**：后端同时接受明文，前端加密失败会自动回退，不会把人挡在门外。

### 限流（Redis + Lua 滑动窗口）
- 注解 `@RateLimit(key, window, limit, dimension)` + 切面；Lua 脚本原子“判断-计数-过期”。
- 已落地：登录 `5 次/60 秒·IP`、注册/手机登录 `5 次/60 秒·IP`、AI 咨询 `10 次/60 秒·用户`；全局 QPS 兜底 `500/秒`（`GlobalRateLimitFilter`，可配 `hda.ratelimit.global-qps`）。超限返回 HTTP 429 / 业务码 429。

### 高并发
- **缓存**：`@Cacheable` 缓存医生详情、科室列表、商品类别（差异化 TTL 打散雪崩）。
- **分布式锁**：`@DistributedLock`（Redisson）加在积分兑换上，锁 `hda:lock:exchange:{userId}`，防并发超卖/重复扣减；锁在事务外层。
- **接口幂等**：`@Idempotent` + 一次性令牌。兑换前先 `GET /api/idempotent/token`，随 `Idempotency-Key` 头提交，重复提交返回 `1201`。
- **异步化**：`@Async` 线程池（有界队列 + CallerRuns + MDC traceId 透传）。AI 咨询的积分发放改为异步，不阻塞应答。MQ 暂不引入，但异步任务已收敛到 `AsyncTaskService`，将来切 MQ 只换实现。

### 数据安全
- **字段加密**：AES-256-GCM 加密落库——就诊记录的 `诊断/处方`、健康档案的 `过敏史/家族史/既往史`。密文带 `ENC:` 前缀，读时对遗留明文向后兼容。
- **出参脱敏**：`@Desensitize` 序列化打码——手机号 `138****8000`、紧急联系电话同理。
- **手机号说明**：`sys_user.phone` 因参与唯一索引与登录等值查询，本轮采用**出参脱敏**而非落库加密（彻底加密需改确定性加密 + 新增 `phone_hash` 检索列，属独立一轮，风险单列）。

### 可观测性（轻量）
- `TraceIdFilter` 为每个请求注入 `traceId`，`logback-spring.xml` 统一日志格式带 `[traceId]`，异步线程透传，全链路可串。错误级别日志单独归档到 `logs/hda-error.log`。

## 四、小程序端（uni-app）

后端全部**向后兼容**，小程序当前通过 HTTPS 明文提交仍可正常登录使用，功能不受影响。若要与 Web 端对齐（RSA 加密 + 双令牌 + 登出），需在小程序内：
1. 引入兼容 uni-app 的 RSA 库（如 `wxmp-rsa` 或纯 JS 实现），登录前调 `/api/auth/public-key` 加密密码；
2. 存储 `refreshToken`，请求 401 时调 `/api/auth/refresh` 换新；
3. 退出时调 `/api/auth/logout`。

这部分我可以在你确认后单独落地。

## 五、验收清单（逐条勾选）

登录安全
- [ ] 正常账密/手机号登录成功，返回含 `token` 与 `refreshToken`
- [ ] 抓包看登录请求，`password` 为 RSA 密文（非明文）
- [ ] access 过期后继续操作，能自动刷新、无需重新登录
- [ ] 登出后用旧 token 调接口 → 被拒（黑名单生效）
- [ ] A 设备登录后 B 设备再登录，A 端下次请求提示“账号已在其他设备登录”

限流
- [ ] 1 分钟内错误登录 6 次 → 第 6 次返回 429
- [ ] 1 分钟内 AI 咨询 11 次 → 第 11 次被限流
- [ ] 压测全局 QPS 超阈值 → 返回 429

高并发
- [ ] 并发兑换同一商品，库存/积分不出现超扣（分布式锁生效）
- [ ] 兑换接口重复提交同一 `Idempotency-Key` → 第二次返回“请勿重复提交”
- [ ] 医生详情/科室/类别二次访问走缓存（观察 SQL 日志不再查库）

数据安全
- [ ] 直接查库，`diagnosis/prescription/过敏史...` 为 `ENC:` 密文
- [ ] 接口返回手机号为 `138****8000` 脱敏形态
- [ ] 应用读取时诊断/处方等能正常解密展示

可观测性
- [ ] 日志每行带 `[traceId]`，同一请求（含异步日志）traceId 一致
