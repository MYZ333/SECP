> 本文档为「先方案后编码」的第一步。请你审阅确认后，我再按此落地代码。

## 一、目标与决策汇总

把当前项目从「能跑的课程设计」升级为「按企业级标准编写的系统」，重点两块：**登录认证安全加固**、**限流与高并发**，并顺带补齐**数据安全**与**基础可观测性**。

下面是我们通过 10 个问题确认的决策，本文档全部据此展开：

| # | 决策项 | 结论 |
|---|--------|------|
| 1 | 登录安全措施 | JWT 刷新令牌 + 黑名单；密码 RSA 传输加密 |
| 2 | 令牌/会话策略 | 单点登录（新登录挤下旧登录） |
| 3 | 限流技术方案 | Redis + Lua 滑动窗口 |
| 4 | 限流覆盖范围 | 登录接口专项、AI 健康咨询接口、全局 QPS 保护 |
| 5 | 高并发手段 | 热点数据 Redis 缓存、分布式锁防重复提交、接口幂等（防重 Token）、异步化（线程池/MQ） |
| 6 | 部署形态 | 单机演示，但**按集群标准编写**（所有状态放 Redis，不用本地内存态） |
| 7 | 可观测性 | 仅结构化日志 + 全局异常（TraceId 链路、统一日志格式） |
| 8 | 敏感数据 | 加密存储 + 接口脱敏（身份证/手机号/病历） |
| 9 | 交付方式 | 先出方案文档（本文），确认后改代码 |
| 10 | 优先级 | 两块均衡全做 |

**一句话原则**：单机部署，但任何"限流/登录态/锁/幂等/缓存"都不能依赖单机内存 —— 全部走 Redis，保证明天直接横向扩成多节点也不用重写。

## 二、现状盘点（改造前必须先认清家底）

难得的是，这套代码底子不错，不少企业级要素已经有雏形，改造要"补齐 + 加固"，而不是推倒重来。

### 2.1 已经具备的能力（保留并强化）

- **单点登录其实已实现**：`AuthServiceImpl.issueToken()` 把每个用户的当前 token 写进 Redis（`hda:login:token:{userId}`），`JwtAuthenticationFilter` 每次请求都拿请求里的 token 和 Redis 里的比对，不一致就当未登录。→ 新登录覆盖旧 token，旧端自然失效，**"挤下线"天然成立**。本次只需在此基础上加"被挤下线的友好提示码"。
- **短信验证码限流是现成范本**：`SmsServiceImpl` 已有三道闸（60 秒重发间隔、单手机号每日 10 次、单 IP 每小时 20 次、连续错误 5 次锁定），全部基于 Redis。**通用限流器会复用这套思路，并升级为 Lua 原子滑动窗口。**
- **改密/重置即失效**：`resetPassword()` 删除 Redis 登录态，旧 token 立即作废。
- **统一返回体与全局异常**：`Result`/`ResultCode`/`GlobalExceptionHandler`/`BusinessException` 已成体系。
- **认证鉴权骨架完整**：`SecurityConfig`（STATELESS + 白名单 + `@EnableMethodSecurity`）、`JwtUtil`、`LoginUser`、JSON 化的 401/403 处理器。
- **空的 `annotation/`、`aspect/` 包**：正好是本次放 `@RateLimit`、`@Idempotent`、`@Desensitize` 及其 AOP 切面的地方。

### 2.2 缺口清单（本次要补的）

| 领域 | 缺口 |
|------|------|
| 登录 | 无刷新令牌（token 过期只能重新登录）；无显式登出 + 黑名单；密码明文经 HTTPS 外裸传（无 RSA） |
| 限流 | 除短信外无通用限流；登录接口无防暴力破解节流；AI 咨询接口无按用户限流；无全局 QPS 兜底 |
| 高并发 | 无声明式缓存（`@Cacheable` 未启用）；无分布式锁（未引入 Redisson）；无接口幂等；AI 等耗时调用同步阻塞，无异步/线程池 |
| 数据安全 | 身份证/手机号/病历明文入库；接口原样返回，无脱敏 |
| 可观测性 | 日志无 TraceId、无统一格式；无慢请求/异常的结构化记录 |

```

   浏览器 / 小程序 ──────┤                                      │
        │ 密码 RSA 加密   └──────────────┬───────────────────────┘
        ▼                                ▼
  ┌───────────────────────── Spring Boot 应用 ─────────────────────────┐
  │  Filter 链                                                          │
  │   ├─ TraceIdFilter        (MDC 注入 traceId，贯穿日志)              │
  │   ├─ GlobalRateLimitFilter(全局 QPS 兜底，Lua 滑动窗口)            │
  │   └─ JwtAuthenticationFilter (access token 校验 + 黑名单比对)       │
  │  AOP 切面                                                           │
  │   ├─ @RateLimit  → 接口级限流 (登录/AI咨询，按 IP/用户维度)        │

  │   ├─ 认证：双令牌签发 / 刷新 / 登出黑名单 / 挤下线                  │
  │   ├─ 缓存：@Cacheable 热点数据 (医生/积分商品/字典)               │
  │   ├─ 锁：Redisson 分布式锁 (积分兑换/写扣减)                       │
  │   └─ 异步：@Async 线程池 (AI 咨询落库 / 通知)                      │

                  ▼                        ▼
         ┌──────────────┐          ┌──────────────┐
         │   MySQL 8    │          │    Redis     │
         │ 敏感字段密文  │          │ 登录态/黑名单 │
         └──────────────┘          │ 限流/锁/幂等  │

                                   └──────────────┘
```

所有跨请求状态集中在 Redis —— 这是"单机写、集群可扩"的关键。

## 四、模块一：登录与认证安全加固

### 4.1 双令牌（Access + Refresh）+ 黑名单

**现状问题**：只有一个 token，过期即掉线；无登出机制。

**方案**：
- **Access Token**：短寿命（建议 30 分钟），承载身份，每次请求携带。
- **Refresh Token**：长寿命（建议 7 天），仅用于换取新的 access token，存 Redis（`hda:login:refresh:{userId}`）。
- **登出黑名单**：新增 `POST /api/auth/logout`。登出时把当前 access token 的 `jti`（或 token 摘要）写入 Redis 黑名单，TTL = token 剩余有效期；同时删除该用户的登录态与 refresh token。`JwtAuthenticationFilter` 增加"黑名单命中即拒绝"判断。
- **刷新接口**：新增 `POST /api/auth/refresh`，校验 refresh token 合法且未被登出 → 重新签发 access（并可轮换 refresh，防重放）。

**改动点**：
- `JwtUtil`：token 加 `jti`（UUID）声明；新增 `generateRefreshToken()`、`getJti()`、`getRemainingMillis()`。
- `JwtProperties`：新增 `refreshExpiration`，`expiration` 缩短为 30 分钟。
- `AuthServiceImpl.issueToken()`：返回 access + refresh；新增 `refresh()`、`logout()`。
- `LoginVO`：增加 `refreshToken`、`expiresIn` 字段。
- `JwtAuthenticationFilter`：命中黑名单（`hda:token:blacklist:{jti}`）直接放行为未认证。
- `SecurityConfig` 白名单加 `/api/auth/refresh`（登出需登录态，不进白名单）。

### 4.2 密码 RSA 传输加密

**现状问题**：前端把明文密码丢进 body，仅靠 HTTPS 保护；抓包/日志/中间人风险。

**方案**（后端已有 Hutool，天然支持，无需新依赖）：
- 新增 `GET /api/auth/public-key`：返回 RSA 公钥（后端持久化密钥对，公钥可公开）。
- 前端登录/注册/改密前，用公钥加密密码字段再提交。
- 后端在 `login/register/resetPassword` 入口用私钥解密还原明文，再走原有 BCrypt 校验/加密逻辑。
- 兜底：解密失败视为参数非法（`ResultCode` 增加 `PASSWORD_DECRYPT_FAIL`）。

**改动点**：新增 `security/RsaKeyProvider`（启动生成/加载密钥对）+ `CryptoService`；`AuthController` 加公钥接口；三个入口 DTO 的密码在 Service 层解密。前端 `web/` 与 `miniprogram/` 登录调用处加加密步骤（用 `jsencrypt`）。

### 4.3 单点登录（挤下线）—— 强化已有能力

**现状**：已天然支持（见 2.1）。**强化**：
- 被挤下线时，返回专门的错误码 `ResultCode.LOGIN_ELSEWHERE`（"账号已在其他设备登录"），前端据此弹窗并跳登录页，而不是笼统的 401。
- 实现方式：`JwtAuthenticationFilter` 里当"token 合法但与 Redis 登录态不一致"时，写一个可识别的响应而非静默放行。

### 4.4 （可选，未选中）登录失败锁定

你本次未勾选"失败锁定 + 验证码"。这里仅备注：如需防暴力破解，最省事的做法是**靠 4.5 的登录接口限流**兜住大部分场景。若后续要账号级锁定，可复用 `SmsServiceImpl` 的 `FAIL_KEY` 计数模式。**本次不实现，等你确认。**

## 五、模块二：限流（Redis + Lua 滑动窗口）

### 5.1 核心：一段 Lua 脚本 + 一个注解

**为什么用 Lua**：把"取当前窗口计数 → 判断 → 自增 → 设过期"做成**一次原子操作**，避免高并发下的竞态（`INCR` + `EXPIRE` 两步非原子会漏限）。滑动窗口用 `ZSET`（member=时间戳，score=时间戳），每次请求先移除窗口外的旧记录，再统计窗口内数量。

`resources/lua/sliding_window_rate_limit.lua`（伪逻辑）：
```
KEYS[1] = 限流 key
ARGV = [窗口毫秒, 阈值, 当前时间戳, 唯一member]
ZREMRANGEBYSCORE key 0 (now-window)   -- 清理过期
count = ZCARD key
if count < limit then
    ZADD key now member
    PEXPIRE key window
    return 1     -- 放行
else
    return 0     -- 拒绝
end
```

### 5.2 声明式注解 `@RateLimit`

放在空的 `annotation/` 包，切面放 `aspect/`：
```java
@RateLimit(key = "login", window = 60, limit = 5, dimension = IP)   // 每 IP 60 秒最多 5 次登录
@RateLimit(key = "ai-consult", window = 60, limit = 10, dimension = USER) // 每用户每分钟 10 次咨询
```
- `dimension`：IP / USER / GLOBAL / SPEL（自定义表达式）。
- `RateLimitAspect` 组装最终 Redis key（`hda:rl:{key}:{dimension值}`），执行 Lua，超限抛 `ResultCode.RATE_LIMIT_EXCEEDED`（HTTP 429）。

### 5.3 三处落地（对应你选的范围）

| 位置 | 维度 | 建议阈值（可调） | 目的 |
|------|------|----------|------|
| `AuthController.login` / `phoneLogin` | IP | 5 次 / 60 秒 | 防登录暴力破解 |
| `ConsultController.chat` | USER | 10 次 / 60 秒 | 控 AI 调用成本、防刷 |
| 全局过滤器 `GlobalRateLimitFilter` | GLOBAL | 例 500 QPS | 系统级兜底防雪崩 |

全局限流放在 Filter 层（比 AOP 更靠前，非登录接口也能拦），登录/AI 用注解精细控制。

### 5.4 新增
- `annotation/RateLimit.java`、`aspect/RateLimitAspect.java`
- `common/ratelimit/RateLimitDimension.java`、`RedisRateLimiter.java`（封装 Lua 调用）
- `security/GlobalRateLimitFilter.java`
- `ResultCode.RATE_LIMIT_EXCEEDED`（429）

## 六、模块三：高并发（缓存 / 分布式锁 / 幂等 / 异步）

### 6.1 热点数据 Redis 缓存

**目标数据**（读多写少）：医生列表、积分商品列表、系统字典类数据。

**方案**：启用 Spring Cache（`@EnableCaching` + `RedisCacheManager`），在 Service 上加：
```java
@Cacheable(value = "doctor:list", key = "#query.hashCode()")
@CacheEvict(value = "doctor:list", allEntries = true)  // 增删改时清
```
- 统一设 TTL（如 10 分钟）+ 随机抖动**防缓存雪崩**。
- 缓存空值（短 TTL）**防缓存穿透**。
- 热点 key 续期 / 逻辑过期 **防缓存击穿**（或用 6.2 的锁重建）。

**新增**：`config/CacheConfig.java`（RedisCacheManager + 序列化 + 各 cache 的 TTL）。

### 6.2 分布式锁（Redisson）防重复提交

**场景**：积分兑换 `PointExchangeService`（防止并发把库存/积分扣成负数）、其他"先查后写"的扣减。

**方案**：引入 `redisson-spring-boot-starter`，关键写操作加锁：
```java
RLock lock = redisson.getLock("hda:lock:exchange:" + userId);
if (lock.tryLock(2, 10, TimeUnit.SECONDS)) {
    try { /* 校验积分/库存 → 扣减 → 落记录 */ }
    finally { lock.unlock(); }
}
```
可进一步封装成 `@DistributedLock` 注解（放 `annotation/`），SpEL 生成锁 key。

### 6.3 接口幂等（防重 Token）

**场景**：兑换、提交健康数据等"点两次会出两条"的写接口。

**方案**：
- 新增 `GET /api/idempotent/token`：发一个一次性 token 存 Redis。
- 客户端提交时带上（header `Idempotency-Key`）。
- `@Idempotent` 切面用 Lua **原子 GETDEL**：存在即消费放行，不存在即判定重复提交（`ResultCode.REPEAT_SUBMIT`）。

**新增**：`annotation/Idempotent.java`、`aspect/IdempotentAspect.java`、`controller/IdempotentController.java`。

### 6.4 异步化（线程池 / MQ）

**场景**：AI 咨询落库、登录积分、（未来）通知推送等非主链路耗时操作。

**方案（单机优先用线程池，接口预留 MQ）**：
- `@EnableAsync` + 自定义 `ThreadPoolTaskExecutor`（`config/AsyncConfig.java`，核心/最大线程、有界队列、拒绝策略、`traceId` 透传）。
- 把 `ConsultController.chat` 里"保存记录 + 发积分"改为 `@Async` 后置执行，主链路只等 AI 结果，缩短响应。
- **单机 MQ 说明**：本次不强上 MQ（避免引入 RocketMQ/RabbitMQ 运维成本），但把异步任务收敛到一个 `AsyncTaskService` 接口，未来切 MQ 只换实现，业务不动。若你坚持要 MQ，我推荐 RocketMQ，另行确认。

## 七、模块四：数据安全（加密存储 + 接口脱敏）

### 7.1 敏感字段加密存储

**目标字段**：手机号、身份证号、病历/诊断内容等。

**方案**：MyBatis-Plus 字段加密。用 `@EncryptField` + `TypeHandler`（AES-256-GCM，密钥放配置/环境变量），写库自动加密、读库自动解密。手机号如需"按号查询"，额外存一列 `phone_hash`（HMAC）做等值检索。

> 注意：现有 `findEarliestByPhone` 按手机号查询 —— 加密后需改为按 `phone_hash` 查。这个改动会触达登录逻辑，实施时单独小心处理 + 数据迁移脚本。

### 7.2 接口出参脱敏

**方案**：`@Desensitize(type = PHONE/ID_CARD/NAME)` 注解 + Jackson 序列化器，返回时自动打码（`138****8000`、`110***********1234`）。对内解密后的明文只在服务端流转，出网一律脱敏。

**新增**：`annotation/Desensitize.java`、`common/crypto/` 下的加解密工具与 TypeHandler、脱敏序列化器。配套 `schema.sql` 字段扩容（密文比明文长）+ 存量数据迁移脚本。

## 八、模块五：可观测性（结构化日志 + 全局异常）

按你的选择，**不上 Prometheus/Grafana**，只做轻量但专业的日志与异常：

- **TraceId 链路**：新增 `TraceIdFilter`，每个请求生成/透传 `traceId` 放入 MDC，`logback-spring.xml` 日志格式统一带上 `[traceId]`，异步线程池透传 traceId。排障时一个 id 串起全链路。
- **统一异常出口**：`GlobalExceptionHandler` 已有，补充：限流 429、幂等重复、解密失败、认证类异常的分类处理，并统一记录 `WARN/ERROR` 结构化日志（含 traceId、userId、uri、耗时）。
- **访问日志切面**（可选）：记录慢请求（如 >1s）与异常请求，JSON 格式，便于日后接入 ELK 不改代码。
- **`logback-spring.xml`**：控制台 + 文件双 appender、按天滚动、级别分离。

## 十一、分阶段实施计划

按"低风险优先、可独立验证"排序，每阶段做完可单独测：

| 阶段 | 内容 | 风险 | 可独立验证 |
|------|------|------|-----------|
| **P0 地基** | TraceId 过滤器、logback、ResultCode 扩充、Redisson/Cache 依赖接入 | 低 | 启动正常、日志带 traceId |
| **P1 限流** | Lua 脚本 + `@RateLimit` + 全局 Filter，落地登录/AI/全局 | 低 | 压测超阈值返回 429 |
| **P2 登录加固** | 双令牌 + 登出黑名单 + 刷新接口 + 挤下线错误码 + RSA | **中**（触认证核心） | 登录/刷新/登出/换设备全流程 |
| **P3 高并发** | `@Cacheable` 缓存、Redisson 锁、`@Idempotent`、`@Async` | 中 | 并发兑换不超卖、重复提交被拦 |
| **P4 数据安全** | 字段加密 TypeHandler + 脱敏 + 存量迁移 | **高**（改存储+迁移，触登录查询） | 新老数据读写正常、出参打码 |

> 建议按 P0→P1→P2→P3→P4 顺序落地。P2 和 P4 触及核心与存量数据，会在动手前再单独跟你确认细节（尤其手机号加密对登录查询的影响）。

---

## 十二、风险与回滚

- **P2/P4 是高危区**：动认证与存量数据。策略——每阶段独立提交、保留开关（如 RSA/加密可配置开关，异常可快速回退明文分支）、迁移前先备份 `hda_db`。
- **前后端联动**：RSA 加密、双令牌、幂等 token 需要前端配合。后端会保持**向后兼容过渡**（如登录同时接受加密/明文一段时间，前端切完再收紧）。
- **单机 → 集群**：本方案所有状态在 Redis，横向扩展无需改代码；唯一注意点是 RSA 密钥对与 AES 密钥需多节点共享（放配置中心/环境变量，不用随机生成）。

---

## 十三、验证与测试计划

- **单元/集成测试**：限流 Lua（并发命中阈值）、幂等（重复消费）、双令牌刷新与黑名单、加解密与脱敏往返。已有 `spring-security-test` 可测认证链。
- **并发压测**：用脚本对 `login`、`consult/chat`、`exchange` 并发打，验证 429、不超卖、幂等。
- **回归**：现有登录（账密/手机号）、改密重置、AI 咨询、积分兑换全流程回归。
- **验收清单**：随代码交付一份"逐条勾选"的验收表。

---

## 十四、需要你确认/拍板的点

1. **令牌时长**：access 30 分钟 / refresh 7 天，是否 OK？
2. **限流阈值**：登录 5次/分·IP、AI 10次/分·人、全局 500 QPS —— 是否按此，还是你有目标数值？
3. **异步**：先线程池、不上 MQ（预留接口），是否同意？若要 MQ 请指定 RocketMQ/RabbitMQ。
4. **数据安全 P4 范围**：手机号加密会牵动登录查询逻辑 + 存量迁移，是否本轮就做，还是先做 P0–P3、P4 单列一轮？
5. **前端**：`web/` 和 `miniprogram/` 的登录改造（RSA、双令牌、幂等 token）是否也由我一并改？

确认后我从 **P0** 开始落地代码。
